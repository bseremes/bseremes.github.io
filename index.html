  • Developpeur Full-Stack

    Dans le monde informatique et particulièrement dans les architectures client/serveur, on dit d'un développeur qu'il est full-stack s'il connaît des stacks complètes du développement d'applications - chaque stack comprenant un frontend, un backend et un système de stockage des données.
    En ce sens il doit connaître au moins :

    • Un langage front comme HTML/CSS/JavaScript, Java pour Android, Objective-C ou Swift pour iOS, permettant généralement un rendu visuel intéractif de l'application en fonction de la plateforme et de la connectivité de l'utilisateur.
    • Un protocole de communication comme HTTP permettant la discussion entre le frontend et le backend.
    • Un langage back comme .Net (ASP, C#), Node.JS, PHP, J2EE ou Ruby, permettant généralement de traiter les requêtes utilisateur.
    • Un protocole de discussion comme des API REST, des ORM ou JDBC entre le backend et les différentes sources de données.
    • Un système de stockage de données comme SqlServer, MongoDB, MySQL, Oracle, Firebase voire Facebook ou Google Documents, permettant de stocker des informations.

    Concrètement, sur un projet de développement d'applications web, mobile et/ou desktop, il est capable est capable d'intervenir sur un ou plusieurs de ces 5 points spécifiques, mais est théoriquement capable de tout développer seul.

    Avec l'arrivé de Node.JS, la multiplication des frameworks front JS (Angular, Ember, Backbone, Cordova, etc...), des API REST de type JSON et des bases de données de type JSON comme MongoDB, de nombreuses combinaisons de stacks sont désormais possibles et on constate une augmentation massive de profils full-stack JavaScript, même jeune, certainement dû au fait qu'il n'y a qu'un unique langage à vraiment maitriser : le JavaScript.

    Et pour ceux qui pensent que les full-stacks sont destinées aux startups, les (vraies) entreprises ayant succédé grâce au Web (Facebook, Google, etc..) énoncent souvent l'hypothèse qu'un développeur qui est conscient des problématiques des stacks complémentaires augmente significativement sa productivité...et celles des autres : ils embauchent principalement des profils full-stacks puis les spécialisent, avec des résultats plutôt satisfaisants, voire mieux que celles qui embauchent directement des experts, types de profil souvent moins flexibles d'entrée de jeu.

    Me concernant, je dirais surtout que cet article est MA définition et les compétences cités en exemples (en gras, et celà s'applique aussi pour la suite...) sont des compétences que je possède réellement.
    Pour l'histoire, j'ai passé ces 4 dernières années majoritairement sur des "stacks web Microsoft" (front : HTML/CSS/JS - back : ASP.Net MVC en C# - base de donnée : SqlServer) mais également sur des "stacks mobile Java" (Java pour Android - JSEE - Oracle) avec quelques projets personnels en stacks web PHP plus classiques (HTML/CSS/JS - PHP - MySql).

    Actuellement, je cherche à me positionner sur un poste de développeur full-stack web avec des frameworks JavaScript (React, Angular, ou Backbone par exemple) en frontend, avec du NodeJS en backend et/ou avec une base de donnée NoSQL comme MongoDB. La raison principale d'utiliser sans modération du JavaScript est qu'après de nombreux tests, voire des débuts de projets, je trouve ces technologies nativement très pratiques pour :

    • le fonctionnement en Agile (SCRUM ou Extreme Programming) et une excellente intégration continue propre aux langages de scripting (contre les langages compilés).
    • le versionning et le branching avec Git et de nombreux services de hosting comme Github ou Stash avec souvent la possibilité de suivre la productivite ou d'automatiser des tâches grâce à un système de hooks...
    • sa légereté et sa simplicité : juste besoin d'un éditeur de texte (comme Atom, Sublime ou Brackets), d'installer Node.JS, de taper npm install et c'est parti...
    • la quantité et le qualité de la communauté JavaScript (et de celles qui sont plus spécifiques) notamment sur Github ou sur StackOverflow.

    De plus, cette compétence se marie à merveille, à mon autre compétence principale de Growth Hacker, me permettant d'améliorer mes méthodes et outils de productivité mais également de mieux comprendre les besoins utilisateurs et donc développer une application qui correspond d'avantage à leurs attentes.
